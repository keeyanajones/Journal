Getting Started

Last updated on April 3rd, 2019

  1. Home
  2. Getting Started

At first glance, Software Engineering looks like learning new computer languages regularly, using logical reasoning, systems understanding, communicating with designers and product manager. However, day to day it is all about solving business problems with technology, designing efficient systems to meet users needs and working with others.

Typically the skills needed include designing and developing software applications, coding, debugging, testing and troubleshooting throughout the application development process. Often a bachelor's degree in Computer Science is required, however, I have found that self taught skills in technology, previous experience building software, and a passion for learning is more important.

The Engineering Process

In my experience, there is no definitive “one size fits all” process for the software engineering process. There are a number of popular approaches based on traditional methods however, when these are applied they often require constant adjustments based on timing, scale, technology, budget and other business factors.

Normally, a combination of traditional approaches in distinct incremental phases is applied simultaneously or iterations until deployment. These phases are usually referred to as the software development life cycle (SDLC). In practice, the approach is selected by the management team in the planning phase of the project.

Thus, an understanding of various process models, techniques, and technologies is essential to the development team. This allows for a cohesive development environment when choosing an integrated development environments, design patterns, paradigms, rules, frameworks, libraries, and models.

The most helpful sub-disciplines in software engineering:

Problem Solving

  1. Probe and Read the problem completely twice
  2. Define the problem completely twice
  3. Determine the problems seriousness
  4. Analyze causes, explore context, and think creatively
  5. Brainstorm possible solutions, Generate, and evaluate
  6. Choose the best solution alternatives and map out support
  7. Solve the problem manually with 3 sets of sample data implementing the solution
  8. Optimize the manual steps
  9. Write the manual steps as comments or pseudo-code
  10. Replace the comments or pseudo-code with real code
  11. Optimize the code

The Object Oriented Analysis and Design Process

The major phases of software development using object–oriented methodology are object-oriented analysis, object-oriented design, and object-oriented implementation.

The Object Oriented Analysis OOA phase of software development, the system requirements are determined, the classes are identified and the relationships among classes are identified. The three analysis techniques that are used in conjunction with each other for OOA are object modeling, dynamic modeling, and functional modeling.

Object–Oriented Design OOD involves implementation of the conceptual model produced during object-oriented analysis. In OOD, concepts in the analysis model, which are technology independent, are mapped onto implementing classes, constraints are identified and interfaces are designed, resulting in a model for the solution.

The implementation details include:

  • Restructuring the class data
  • Implementation of methods, controls, and associations

OOD is a method of design encompassing the process of object-oriented decomposition and notation for depicting the logical and physical; static and dynamic models of the system design. Important topics during OOD also include the design of software architectures by applying architectural patterns and design patterns with OOD principles.

Quality Factors Classification

My model classifies all software requirements into 13 software quality factors. The 13 factors are grouped into four product categories – operation, revision, transition, and other factors.

  • Operation Factors − Correctness, Reliability, Efficiency, Integrity, Usability
  • Revision Factors − Maintainability, Flexibility, Testability
  • Transition Factors − Portability, Reusability, Interoperability
  • Other Factors - Understandability, Conciseness and Consistency
